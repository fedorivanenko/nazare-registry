# Fast Theme Dev Asset Watcher

## Goal

Make local theme development responsive when many Nazare components are installed.

The current `vite build --watch` path uses Rollup build watch. It still walks the full build graph, prints the full output list, and may rebuild many chunks when one Liquid, CSS, or JavaScript source changes. The dev watcher gives Nazare a development-only pipeline that keeps compiler state warm, emits fewer development assets, and writes only changed files for Shopify CLI to sync.

---

## Scope

Included:

- new development asset watcher script at `theme/default/nazare/dev-assets.js`
- `package.json` script wiring:
  - `dev:assets`: runs the dev asset watcher
  - `dev`: runs `dev:assets` beside `shopify theme dev -e development --nodelete`
  - `build`: remains production `vite build`
- reuse of existing theme scanner behavior for Liquid/static render graph validation
- development JavaScript bundling through esbuild context/watch APIs
- development CSS output as one shared file: `assets/sections.dev.css`
- development generated `snippets/section-css.liquid` that loads `sections.dev.css` for any section name
- development generated `snippets/section-css-preloads.liquid` with no per-section preload output
- idempotent writes for generated Liquid/config/runtime intermediates and final assets
- rebuild logging that reports only changed input path(s), rebuilt output path(s), warnings, and elapsed time
- tests for script wiring, generated dev snippets, no section-group overwrite, and idempotent writes

### Dev command surface

```json
{
  "scripts": {
    "dev": "sh -c 'npm run dev:assets & assets_pid=$!; npm run dev:theme; status=$?; kill $assets_pid 2>/dev/null; exit $status'",
    "dev:assets": "node nazare/dev-assets.js",
    "dev:theme": "shopify theme dev -e development --nodelete",
    "build": "vite build",
    "watch": "node nazare/dev-assets.js"
  }
}
```

### Dev output files

The watcher writes development-friendly outputs:

```txt
layout/theme.liquid
config/settings_schema.json
config/settings_schema.json.nazare
scripts/theme.js
snippets/section-css.liquid
snippets/section-css-preloads.liquid
assets/theme.js
assets/<lazy-js-chunk>.js
assets/sections.dev.css
```

Production `vite build` still writes production CSS split by section (`assets/<section-name>.css`) and production JavaScript chunks according to `vite.config.js`.

## Success behavior

- `npm run dev:assets` starts a long-running watcher and prints a ready message with watched source roots.
- Initial watcher pass validates Liquid/render/module graph using the same scanner rules as production.
- Initial watcher pass writes required generated Liquid/config/runtime files and dev assets.
- JavaScript changes rebuild through esbuild context/watch without invoking Rollup/Vite production build.
- Liquid/snippet/config/layout changes rescan the graph and regenerate only changed generated files.
- Development CSS is emitted as `assets/sections.dev.css` and includes Tailwind utilities for all layout, section, snippet, and Liquid-filter classes scanned by Nazare.
- Generated `snippets/section-css.liquid` in dev mode loads `sections.dev.css` at most once per page, even when many sections call the snippet.
- `sections/header-group.json` and `sections/footer-group.json` are never generated, rewritten, watched as owned output, or deleted by the dev watcher.
- Console output is compact, for example:

```txt
[nazare] dev assets ready in 812ms
[nazare] sections/s-hero.liquid -> assets/sections.dev.css, snippets/section-css.liquid in 143ms
[nazare] scripts/sections/s-menu.js -> assets/theme.js, assets/s-menu.js in 58ms
```

- Unchanged generated content is not rewritten, so Shopify CLI does not resync files whose bytes did not change.
- `npm run build` behavior is unchanged and remains the production verification path.

---

## Failure behavior

- Invalid CSS directive fails the watcher pass with a clear error and keeps the process alive for the next file change.
- Invalid layout directive fails the watcher pass with a clear error and keeps the process alive for the next file change.
- Missing snippets, render cycles, invalid `data-nazare-use`, and missing JS modules are reported with the same error text as production scan where possible.
- esbuild bundle failures are reported without deleting previous working assets.
- Tailwind/dev CSS failures are reported without deleting previous working CSS.
- If an output path cannot be written, the watcher reports the path and error message, then waits for the next change.
- Startup failure due to missing required dependency exits non-zero and names the missing package.
- Failure cases must not mutate unrelated user files and must not rewrite Shopify/admin-owned section group JSON.

## Architecture notes

The watcher is development-only. It should optimize for fast local feedback and low Shopify CLI sync churn, not byte-for-byte parity with production asset splitting.

Use existing scanner/parser behavior from `theme/default/nazare/vite-plugin.js` rather than duplicating directive, render graph, and `data-nazare-use` validation. If necessary, refactor shared scanner/generator helpers into support modules under `theme/default/nazare/` while keeping the public Vite plugin export stable.

JavaScript should use esbuild `context()` so rebuilds reuse graph/cache state. The generated runtime should keep the same public behavior (`initNazare`, `destroyNazare`, Shopify section events, lazy module loading), but the development bundler may emit a different chunk graph than Vite production.

CSS should use a single development CSS output to avoid dozens of Rollup/Tailwind section entries during watch. The dev CSS source list should include layout, all sections, all snippets, and inline Liquid-filter classes found by scanner helpers. More CSS on the page is acceptable in development; production keeps section-scoped CSS.

The dev watcher should use a small debounce window for bursty editor writes. It should compute affected output categories from changed paths:

| Changed path | Work |
|---|---|
| `scripts/**/*.js` | esbuild JS rebuild |
| `sections/**/*.liquid` | rescan graph, regenerate dev CSS/snippets/runtime inputs as needed |
| `snippets/**/*.liquid` | rescan graph, regenerate dev CSS/snippets/runtime inputs as needed |
| `layout/theme.source.liquid` | regenerate `layout/theme.liquid` |
| `config/*.settings.json` or `config/theme.settings.json` | regenerate settings schema |
| `styles/base.css` | rebuild base CSS through existing/prod path or document as production-only if not supported in v1 |

Development output should use `writeText`-style idempotent writes. This matters because Shopify CLI watches the theme tree; writing unchanged files causes needless syncs and reloads.

Section group files are intentionally not part of the watcher output graph. They are scaffold/admin-owned files. Shopify admin may update them, and neither Vite nor the dev watcher should reconcile their section membership or ordering.
